# Article Pitch: Building a GitHub Issue Scout with Crawlee

## Working Title

Building a GitHub Issue Scout with Crawlee and Human-in-the-Loop Triage

## Short Pitch

Open-source bounty hunting looks simple until you inspect the actual issues. Many paid-looking tasks are already crowded, reserved for maintainers, core-team-only, stale, or full of low-quality automated attempts.

This article shows how to build a small Crawlee-based scout that searches public GitHub issues, extracts structured paid-work signals, flags risky patterns, and produces a human-reviewable shortlist before anyone writes code or comments on an issue.

Working demo repository: https://github.com/mysubb01/apify-github-issue-scout

Portfolio page: https://mysubb01.github.io/apify-github-issue-scout/

The repository is also Apify Actor-ready with `.actor/actor.json`, `.actor/input_schema.json`, `Dockerfile`, and a local `npm run actor:local` command.

Full draft: https://github.com/mysubb01/apify-github-issue-scout/blob/main/article-draft.md

## What Readers Will Build

- A Crawlee crawler that queries public GitHub issue search results.
- A simple typed data model for issue leads.
- Scoring rules for reward signals, helpful labels, and low-competition issues.
- Risk filters for maintainer-only, core-team-only, star-gated, or crowded issues.
- A JSON output that can be reviewed manually or used in a later workflow.
- An Actor-ready adapter that writes summary output and dataset records.

## Why It Fits Apify/Crawlee

- It is a real developer workflow, not marketing copy.
- It uses Crawlee as the collection layer and keeps the output structured.
- It teaches readers how to turn noisy public web data into a safer operational decision.
- It includes working code and an honest discussion of trade-offs.

## Proposed Outline

1. The problem: bounty boards and GitHub search are noisy.
2. The data model: what a useful issue lead needs.
3. Crawlee setup: fetching public GitHub search result pages/API responses.
4. Signal extraction: labels, comments, reward text, recency.
5. Risk filters: maintainer-only, core-team-only, star-gated, and crowded issues.
6. Ranking output: `pursue`, `review`, and `skip`.
7. Human-in-the-loop rules: never auto-comment, always verify locally first.
8. Next steps: adding Playwright page inspection, notifications, or Apify Actor deployment.
9. Packaging the same workflow as an Apify Actor with input schema and dataset output.
